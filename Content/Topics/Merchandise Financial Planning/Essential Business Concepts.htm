Planning With VAT

If your organization includes VAT (value added tax) in its planning process, there are a few required configuration steps that must be completed prior to beginning to plan:

  1. Create a verbose product hierarchy: in order for VAT calculation functionality to work properly, a verbose product hierarchy is needed. This basically just means that data is able to be stored at all levels of the hierarchy rather than just the lowest leaf levels.
    For example, if my hierarchy consists of the levels Company > Gender > Class, a verbose hierarchy allows data to also be stored against the Company and Gender levels, whereas a regular hierarchy would only allow data at the Class level. Because VAT rates can be set at any level of a product hierarchy, a verbose hierarchy is needed to allow the system to read the correct rates when calculating VAT related metrics.
    To do this:
    Click Plan > Views then locate and open the Hierarchy Filtering view:

    Next, select the Variant entity type and click Execute:

    If you need to create a new hierarchy, click New, enter a Hierarchy Name and Description, check the Allow Data Storage at All Levels checkbox, then Save:

    Lastly, set the Order Number and Enabled checkbox for the desired Item Master Groups and Save:

    Note: If a hierarchy already exists that you would like to use for this purpose, you can simply select the hierarchy, check the Allow Data Storage at All Levels checkbox and Save
    Important: check with your JustEnough technical representative before making this change to hierarchies that are used in modules other than Assortment Planning and MFP. Other modules, such as Allocation and Group Forecasting are not designed to work with verbose hierarchies.
  • Set the MFP product hierarchy VAT rates will be assigned to: the next step in setting up MFP for planning with VAT is telling the system which hierarchy to use for getting the VAT rates to be applied to your plans.
    To do this:
    While still in the Hierarchy Filtering view, click Fields in the Hierarchy Configuration section and drag the Hierarchy ID field into the grid:

    Note the Hierarchy ID for use in the next step:

    Next, click Maintain > Policies, select the Planning VAT Rate Product Hierarchy ID from the Module: Financial Planning folder at the left, enter the Hierarchy ID you noted in the last step in the New Setting field, and click Apply Changes:

  • Enable VAT in MFP: the final step is to enable the use of VAT in MFP. To do this, while still in the Policies module, select the Enable VAT in MFP policy from the Module: Financial Planning folder at the left, select True in the New Setting field, and click Apply Changes:

    Note: like all system policies, the Enable VAT in MFP policy is a global setting, meaning all MFP plans will have VAT enabled once this policy is set to True. That said, if you have a VAT rate of zero set for the product/site levels included in a given plan, ex VAT metric values will be identical to their regular counterparts and there will be no mathematical effect on the plan.

Assuming you have also set up VAT rates in the Planning VAT Rates view, after the above steps are complete, you will be able to plan with VAT included in MFP. For more information on setting VAT rates, please refer to Currency and VAT Management > Essential Business Concepts > Planning VAT Rates.

Creating New Plans

When working in the financial plan maintenance view, the first action our clients need to take is to create a master plan.  A master plan is a Financial Plan that is created in the MFP module and is not editable.  This plan is considered the “single” version of the truth.  Master plans should be used as approved plans within the planning process.  Master plans contain historic and plan periods.  Historic periods represent “what actually happened” within the business and plan periods are the approved plans for future periods.  In managing a Master plan it is recommended that a super user or system administrator has control over this feature within the system.  To create a master plan see the below example.

To create a master plan within the financial plan maintenance view, click on the “New” icon located in the top left portion of this view.  Once the user has clicked on the “New” icon, the maintain financial plan window will appear on screen.

The first tab a user works with in this view is the General tab.  The general tab is where the user will give the master plan a name and description, define the plan type, the currency to work with, how the plan will be seeded, the calendar to plan with, the level of the calendar, and finally the start period.  These general setting form the basis of the structure of the master plan. 

Name:  This is free form text and describes the name of the plan

Description:  This is free form text and provides the opportunity to describe what the plan is and any other important information about the plan that may be important to the user group.

Plan Type:  The plan type defines the metric set and calculations that will make up the master plan.  There are three main plan types to choose from.

Location Plan: This plan provides the ability to plan by location at defined levels of the product hierarchy.  This plan contains the ability to plan sales, margin and Open to Ship to support the location planning process.

Hybrid Cost-Based Plan: This plan provides the ability to plan by Channel/location groups at defined levels of the product hierarchy.  This plan contains the ability to plan sales, cost of sales, margin, inventory, receipts, and open to buy to support the budgeting process.  This plan is for clients who value their inventory at cost.

Hybrid Retail-Based Plan: This plan provides the ability to plan by Channel/location groups at defined levels of the product hierarchy.  This plan contains the ability to plan sales, cost of sales, margin, inventory, receipts and open to buy to support the budgeting process.  This plan is for clients who value their inventory at retail.

Hybrid Cost Dual-Markdown Plan: This plan type is for organizations who wish to plan their permanent and promotional markdowns separately. Promotional markdowns are tied directly to Promotional Sales and reduce AUR for periods in which they are realized, impacting margin through that reduction. Conversely, because markdown sales do not necessarily occur in the same period in which the actual markdowns are taken, Permanent Markdowns are managed strictly as a percentage of sales, and are subtracted directly from the Maintained Margin value for the period in which they occur. In this plan type, average unit cost (AUC) and average unit ticket (AUT) do not typically change from period to period because of changes to the sales plan. AUC will change as a result of incoming receipts or other changes to the inventory mix. AUT will only change if the user changes it directly or indirectly, through a change in IMU %.

Plan Currency:  This selection defines the base currency in which the users will be working with, all currencies are supported in this version.

Management Type: determines whether a plan will be created within JustEnough (Manual type) or data for the plan will be imported via API (Import type); set to Manual by default

Error: if an import post runs for a plan of type Manual, the plan will not be affected and an error will be raised

Note: when import posts run for plans of type Import, hierarchy levels will automatically be added to the plan based on the import data; if desired, the API can be run in “Window” mode, deleting all existing data in a plan and re-loading it from the import data

Seed Type:  This selection defines how future periods in the plan will be seeded for the metrics offered within the plan type that has been selected.  There are three standard seeding options to choose from.  Seed Types in our standard offering cannot be combined to seed future periods when creating the plan.

Historic Sales LY:  Selecting this option will seed future periods with sales, sales units, and average unit retail from last year’s comparable periods.

Historic Sales LLY:   Selecting this option will seed future periods with sales, sales units, and average unit retail from two years ago comparable periods.

Forecasted Sales:   Selecting this option will seed future periods with sales units that are coming from the forecast module.  Sales are derived by multiplying the forecasted sales units times the average unit retail from last year’s comparable periods.

Important: regardless of the selected Seed Type, metrics that would be seeded with negative values based on the available data indicated by that type will be seeded with zeroes instead

Calendar:   This selection defines the calendar that users will be working with in the master plan.  We offer two standard selections based on a retail planning calendar.  Users can select a 4-4-5 calendar or 4-5-4 calendar.  These calendars are NRF retail planning calendars.  It is possible to import custom calendars during implementation of this module if necessary.

Calendar Level:  This selection defines the lowest level of time that users will be working with in the master plan.  There are three standard options a user can select.  (Year, Month, Week)  Most plans created in this module go to the Month or Week level.

Start Period:  This selection defines the first period contained within the plan.  This can be a historical period or future period.

Stock Split Supply Chain: The supply chain used to determine how inventory is split among DCs. (The Supply Chain Status field indicates whether the supply chain is valid for use with this plan.)

Stock Split Sales Types: The sales types to consider when apportioning stock to DCs.

Supply Chain Status: This read-only field indicates whether or not the supply chain is valid. (See Validating a Supply Chain for more information.)

Owner:   This will be greyed out until the plan is created.  Once the plan is created the system will display the name of the user who created the plan.

Creation Date:  This will be greyed out until the plan is created.  Once the plan is created the system will display the date the plan was created.

Status:   This will be greyed out until the plan is created.  Once the plan is created the system will display the status the plan is in.  Plan status will be covered in the Plan approval section of this document.

The second tab a user works with in this view is the Product tab.  The Product tab is where the user will define the product hierarchical structure contained within the plan.  The product hierarchy levels selected define the levels in which users can work with.

The top window of the screen is where the plan levels are selected.  In the example above, the lowest level of this master plan will be category.

The bottom window of the screen is where the user will determine which categories will be contained within the master plan.

 

The third tab a user works with in this view is the Channel tab.  The Channel tab is where the user will define the location hierarchical structure contained within the plan.  The location hierarchy levels selected define the levels in which users can work with.

The top window of the screen is where the plan levels are selected.  In the example above, the lowest level of this master plan will be channel.

The bottom window of the screen is where the user will determine which channels will be contained within the master plan.

The fourth tab a user works with in this view is the User Access tab.  The User Access tab is where the user will define the user/role access permissions within the plan.  Access permissions can be defined by role within the organization or by User.  In the bottom window of this view, setting up the permissions requires that the user navigate through three tabs.  (Product, Channel, Merge Measure Set).  In the example above the highlighted user has read and write access to Women’s Denim.  This means that the user can view and edit Women’s Denim for any branch created that is connected to this master plan.  For the master plan, the user will have read and merge rights for Women’s Denim. 

Access Permissions are also defined by channel.  The user can click on the channel tab and define the level of the location hierarchy roles and users have for read and write access.  In the example above the highlighted user has read and write access to US Retail and E-Com channels.  This means that the user can view and edit Women’s Denim for the US Retail and E-Com for any branch that is connected to this master plan.  For the master plan, the user will have read and merge rights for Women’s Denim.

Access Permissions control which measure sets can be merged into the master plan from a branch that belongs to that master plan.  Measure sets are defined in the Header Area controls section.  In the example above the highlighted user has permission to merge all measures within the Hybrid Cost-Based Plan type into the master from the branch or branches they are working with.

The last tab a user works with in this view is the View Collection tab. The View Collection tab defines the collection of metrics that are contained within the plan as well as the collection of metrics that are contained within any snapshot taken of the plan.  View collections are set up by clicking add for both the top and bottom window in the view.  Upon clicking add, the user will select the Plan Type and View Collection set that is to be associated with this plan.  Once this screen is completed, the user will click OK and that will trigger the system to create the plan based on all the criteria that was defined in the five tabs contained in the maintain financial plan window.  Upon clicking OK, the system will return the user back to the Financial Plan Maintenance view and will display the plan that has been created.

  • Note: plan view collection sets defined in this tab are inherited by default by any Branch plans created from a Master plan. If the same view collection sets assigned to a Master plan are desired for use in Branch plans beneath it, this section can be left blank. Only if different view collection sets are desired must the user make changes to this section.

Note: The Master Financial Plan that has been created will update to “Created” state.  This means the plan is ready to be initialized.  Initializing the master plan basically populates all the measures defined for historical and future periods.  A plan must be initialized in order to view and work with the Master Plan.  To initialize a plan the user will simply click the Initialize icon in the header of the Financial Plan Maintenance view.  Once the user clicks initialize a process will run and return the user back to the Financial Plan Maintenance view and the Master Financial Plan will be assigned the status of Initialized.

Branch Plans

When working in the financial plan maintenance view, after the Master Plan has been initialized, users will have the opportunity to create Branch plans.  Branch plans are a copy of the master plan and can be edited by users who have permission access.  To create a Branch plan, the user will follow the same steps as creating a master plan.  A branch plan is a Financial Plan that is created from a Master plan in the MFP module and is directly editable.  The branch plan is used to make changes to a financial plan.  Branch plan examples include “working” plans, “stretch” plans or “shrink” plans.  There is no limit to the number of branches that can be created.  A branch will always be associated with a Master plan.  Once a branch plan has been created it will show up underneath the Master plan it is associated with in the financial plan maintenance view. 

As mentioned above, to create a branch plan a user will have to go through the same steps as creating a master plan, however all of the criteria in each of the tabs will be pre-populated with the same selection criteria of the master plan.  To create a branch plan, the user will highlight the master plan they want to create a branch from and then click on the Branch icon.

Once the user has clicked on the Branch icon, the Branch Financial Plan window will appear.

The user has the ability to give the branch a name, description and start period and end period in the first tab that appears.  The two most important tabs when creating a branch are the product and channel tab.  Those tabs will contain the same selections as the master plan, however, the user has the ability to narrow down the selections.  Once the branch has been defined the user will click ok and then initialize the branch in the Financial Plan Maintenance view.

Once the branch has been initialized, users can open up the branch and begin making changes to the plan.  To open a branch or master plan, the user will highlight the desired plan and click OK.

Snapshots

Snapshots serve as the third main concept of plans and can be managed in the Financial Plan Maintenance view.  Snapshots are locked down versions of master or branch plans.  The main differences between a snapshot and any other plan is they do not actualize as we move forward in time.  Snapshots can be accessed on the right hand side of view and will be specific to the master or branch plan highlighted on the left hand side of the view.  To access the snapshot functionality the user will click on the snapshots tab on the left hand side of the view.

Snapshots Tab:  Launches the Manage Snapshots dialog - this is used to set up plan snapshots which occur on a regular schedule as well as to create and delete snapshots manually.  Snapshots are, as their name implies, captures of the state the current plan at the time they are generated for the purpose of future comparisons as users work with the plan. Snapshots can be used within the main planning view, the comp/non-comp tab and the sales by type tab within Merchandise Financial planning; for details on the metrics included in snapshots, see Comparison metrics listed in under the respective sections for these supporting views.  There are two main components that make up the Snapshots tab.  The top portion of the view is the MFP Snapshot Templates.

  • Snapshot Templates: list of all snapshot templates for the current plan, which the current user has access to view; used to create recurring snapshots of the current plan
  • New: adds a new snapshot template line
  • Delete: deletes the currently selected snapshot template(s) from the plan
  • Fields: opens the fields Customization dialog; users can drag fields to/from the snapshot templates column header area and this dialog to customize the view
  • Custom layouts can be saved/restored by right-clicking in the Snapshot Templates area and selecting Save Layout or Restore Layout, respectively
  • Name: user editable alpha numeric; name for the snapshot template
  • Description: user editable alpha numeric; description for the snapshot template
  • Remove By Days: user editable positive integer; number of days from the generation of snapshots created by a template after which the system will automatically delete them; used for maintenance purposes
  • Snapshot Start Days: user editable positive integer; number of days backward from the date a snapshot is generated to be captured
  • Snapshot End Days: user editable positive integer; number of days forward from the date a snapshot is generated to be captured
  • Product Level: user editable dropdown; selection of a product hierarchy level to include in snapshots generated by the template; available choices based on the assigned product hierarchy for the current plan
  • Channel Level: user editable dropdown; selection of a location hierarchy level to include in snapshots generated by the template; available choices based on the assigned location hierarchy for the current plan
  • Schedule: user editable dropdown; sets the frequency and timing of snapshots generated by the template; standard options are:
  • Weekly Mondays
  • Monthly - first of month
  • Every 15 Days
  • Note: the exact timing produced by each choice is dependent on the calendar assigned to the current plan
  • Snapshot Access: user editable dropdown; sets user access to snapshots generated by the template; public, private, or shared
  • Snapshot Users: user editable multi-select; if Snapshot Access is Shared, allows selection of specific users to have access to snapshots generated by the template
  • Template Access: user editable dropdown; sets user access to the template; public, private, or shared
  • Template Users: user editable multi-select; if Template Access is Shared, allows selection of specific users to have access to the template
  • Apply: applies user changes made to snapshot templates and saves newly created templates since the last Apply
  • Cancel: cancels user changes made to snapshot templates and removes templates created since last Apply

The bottom portion of the view is the MFP Snapshots


  • Snapshots: list of all snapshots for the current plan, whether created manually or by a template, which the current user has access to view
  • New: adds a new snapshot line to the list
  • Delete: deletes the selected snapshot(s) from the plan
  • Fields: opens the fields Customization dialog; users can drag fields to/from the snapshots column header area and this dialog to customize the view
  • Custom layouts can be saved/restored by right-clicking in the Snapshots area and selecting Save Layout or Restore Layout, respectively
  • Start Date: non-editable calendar control; sets the start date for the snapshot window of time
  • Notes: must be set before End Date; field locks after snapshot is generated
  • End date: non-editable calendar control; sets the end date for the snapshot window of time
  • Notes: must be set after Start Date; field locks after snapshot is generated
  • Remove By Date: user editable calendar control; sets the date on which the system will automatically delete the snapshot; used for maintenance purposes
  • Note: must be set after the current date
  • Product Level: non-editable dropdown; selection of a product hierarchy level to include in the snapshot; available choices based on the assigned product hierarchy for the current plan
  • Note: field locks after snapshot is generated
  • Channel Level: non-editable dropdown; selection of a location hierarchy level to include in the snapshot; available choices based on the assigned location hierarchy for the current plan
  • Note: field locks after snapshot is generated
  • Access Level: user editable dropdown; sets user access to the snapshot; public, private, or shared
  • Access Users: user editable multi-select; if Snapshot Access is Shared, allows selection of specific users to have access to the snapshot
  • Created From Template: read-only alpha-numeric; name of the template the snapshot was generated from; NULL if manually generated
  • Apply: applies user changes made to snapshots and generates newly created snapshots since the last Apply
  • Cancel: cancels user changes made to snapshots and removes snapshots created since last Apply
