Practical Application

Below are some tips and examples to help the user get the most out of the available functionality in the Plan creation and Plan management sections of this module.

  • Master/Branch Plan set up tips:
  • Master plan should contain all necessary product and location hierarchical components to properly manage an OTB process.
  • Stay at a level that is reasonable in terms of the layers available to plan.  When managing an OTB think of the spending on receipts and how to manage that spend at an executive type level.  Executives don’t want information that is too granular, so keep it at a level that is effective and everyone in the organization can understand.  An example would be planning at a category level or layer below division.  If total Female was a division and underneath that layer is categories of product such as denim, skirts, dresses, blouses, etc.  It’s very easy to see where the growth and declines of the business are occurring at that level.  Spend the time managing the OTB at that level in order to move inventory investment into the right categories at the right time.
  • Create a master working branch underneath the master plan.  A master branch simply contains everything in the hierarchical structure for product and location.  This becomes the branch that everyone moves their changes into.  This allows for a holistic review of the business prior to approving a plan to merge into the master.  Users can then create their own branches off the master branch and merge their changes into the master branch.  Ensure the end date of this branch goes out as far as possible into the future.
  • When setting up read/write access for users, consider how much data you want them to see vs. how much data you want them to edit.  An example would be to give the Women’s denim planner write access for that category, but read access for total women’s.  They have the ability to make changes to Denim, but can also understand what is happening in total Women’s.
  • When creating branches to edit the plans and there is a formal plan approval process in place, branch off the time periods that should be reviewed and approved.  This will streamline the process and is very helpful when conducting in-season plan reviews.  Additionally, upon approval, only the changes to those time periods will be merged to the parent branch or master plan.
  • Snapshot Utilization: utilize snapshots as part of the plan review and approval process.  When a plan has been approved, take a snapshot so that there is record of the plans at that point in time.  It will make it easier for the users to understand Plan vs. Actual and the variance between the two
  • Use Snapshot Templates to Schedule Regular Snapshots: snapshot templates can be created to automatically take snapshots on a regular schedule. Schedule snapshots to be taken of the Master plan and Master working branch every week.  That will make reviewing the business simple and easy as the comparisons of plan vs. actual on a weekly basis will be available directly in the main planning view
  • Set an Appropriate Calendar Level: if week to week planning is an important part of the business, the calendar level should be week.  Otherwise select Month as the calendar level. 
  • Create User Specific Master and Branch Plans: be cognizant of the intended users when creating branch and sub-branch plans. Creating branches that only contain the product/site levels each user is responsible for will speed the planning process by limiting the amount of data the system has to read and process for each transaction. If your organization includes planners that are specifically concerned with higher levels of the product/site hierarchies, you can also create additional Master plans for them that don’t include the lower levels of the hierarchy. These plans can ultimately be reconciled with lower level plans via Rebalance functionality.
  • Adding New Leaf Levels to Existing Plan Hierarchies: as your business evolves, you will naturally want to add new product/site levels to your existing plans. Once the new leaf levels are created within the desired ItemMaster/Site Group(s), first use the Initialize functionality on the Master plan, selecting ONLY THE NEW LEVEL(s) to be added. Once complete, you can now use the Reseed functionality on the branch plans beneath that master, again selecting ONLY THE NEW LEVEL(s) that were added to the master. In this manner, you can set up new levels in your existing plans without losing the data for the existing levels
  • Note: “leaf” level refers to an additional entity within an existing ItemMaster/Site Group. For example, say SiteGroup 4 is Site and contains the sites Tempe, Flagstaff, and Phoenix. If I add a new entity called Sierra Vista for a new site that is opening, this is adding a new leaf level to SiteGroup 4. The above example does not pertain to adding a new structure level. An example of adding a new structure level would be if I have a Site Hierarchy consisting of SiteGroups 1-4: Company, Country, District, Site and then I want to insert a new SiteGroup – Region – in between Country and District. This type of change constitutes a hierarchy restructure and requires that a new Master plan be created.
