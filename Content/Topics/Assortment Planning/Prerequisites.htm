Prerequisites

Before creating assortment plans, the following steps must be completed:

  1. Definition of product and site hierarchies: Plan > Views > Maintenance > Hierarchy Filtering
  2. Optional: If you desire to use product and site attributes for the creation of site clusters and/or within the assortment planning process, Attribute Sets, Families, and Values must be defined, and at a minimum, at least one set must be assigned to the topmost level of both the product and site hierarchy: Maintain > Attribute Management
  • Note: site and product attribute associations can be set via your configured data import processes, but they can also be managed manually in the following places:
  • Site Attributes: Manage > Site Manager > Site Attribute Management
  • Product Attributes: Manage > Variant Manager > Variant Attribute Maintenance
  1. Optional: If you will be managing the sales plans of ongoing items through the Forecasts module (replenishment forecast integration), it is recommended that you homogenize the dates for the time buckets used in forecasting and the periods to be used in Assortment Planning; it is also recommended that you review the information for Replen Status items in this manual (Line Visualization > Placeholder & Collection Modes > Main Planning View > Status) to familiarize yourself with the specifics of how such management works between the two modules
  2. Creation, snapshotting, and validation of cluster sets to used: Plan > Cluster Site Manager
  • Note: cluster set validations occur automatically on plan creation as well as when cluster sets are snapshotted.
  1. Definition of Currency Exchange Rates, VAT Rates, and Price Translations
  • Currency: Plan > Views > Currency > Currency Exchange Rates
  • VAT: Plan > Views > Currency > Planning VAT Rates
  • Price Translations: Plan > Views > Currency > Price Translation Management and Price Translation Hierarchy Assignment
  • Note: users not requiring VAT and/or Price Translations functionality must still set a rate/factor of 1 (indicating no change) and assign it for all site/product hierarchy levels, as these values are utilized in many system calculations within Assortment Planning
  1. Review/set foundational system policies: more system policies exist than are listed here, but the following are related to foundational data handling within Assortment Planning. Those not listed here are discussed in the specific sections they pertain to. All can be found in Maintain > Policies > Module: Assortment
  • Assortment days history to actualize: The number of days back to go when actualizing sales, receipts and stock on hand for assortment planning; default is 60
  • Assortment days history to maintain: The number of days back to maintain detail data for in a live assortment plan. Data older than this date will be summarized and then deleted.  Entering the value "0" will keep all history; default is 800
  • Important: this policy is useful for database size management, however it should be noted that the amount of historical data maintained has a direct correlation with the data that can be used to seed historical placeholder/variants for in the Seed process (see Header Area Controls > Seed later in this section for details) and the date from which to begin seeding cannot exceed the date designated by the current date minus the value set for this policy
  • Assortment Order Type: Variant Orders created from assortment planning are reconciled to individual clusters when coming back as confirmed orders from the host system. Inventory Orders uses a calculated split factor to reconcile inbound order quantities to clusters (see Line Visualization > Supporting Views > Receipt Plan View for details); default is Inventory Orders
  • Days to keep deleted data: The maximum number of days that deleted data will be held (hidden) in the database.  Set a lower number for more aggressive deleting and more space conservation; default is 30
  • Default Commitment Merge: sets the default merging behavior of order lines when commitment orders are created: 0 = None, 1 = Merge by Variant (see Commitment Orders in Order Manager > General Functionality > Creating Commitment Orders for details); default is 1 (merge by variant)
  • Receipt Cluster Merge All: When commitments are generated, whether to group receipts for variants together across clusters by default, or whether to keep them separate by default.
