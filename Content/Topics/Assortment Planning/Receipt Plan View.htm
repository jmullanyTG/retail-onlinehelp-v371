Receipt Plan View

Receipt Plan view allows users to hone in on the ideal receipt strategy for items they are planning to purchase. Set constraints, such as minimum Periods of Supply, define the periods in which receipts are allowed to arrive, and how many receipts are needed. From here, you can also send commitment orders to JustEnough Order Manager for final processing, review, approval, and submission to external systems.

  • Execute: refreshes the view
  • Note: this is not typically necessary, as the view refreshes on its own when a new placeholder/collection is selected in the main planning view, when changes are applied, and when the View Mode is changed
  • View Mode: user editable dropdown; selects the view mode for receipt plan view – Total Company or Average Store
  • Total Company: allows the user to view/plan revenue and unit metrics as total company values – meaning the total volume for all sites contained in whatever level the Pivot Grid is aggregated to
  • Average Store: allows the user to view/plan revenue and unit metrics as average store values – meaning the per site volume for the number of sites contained in whatever level the Pivot Grid is aggregated to
  • General Concepts
  • Editable Metrics: denoted by light blue row coloration, these are the only metrics that are editable by the user in Receipt Plan view; all others are calculations or reads from other sources
  • Note: Changing Minimum Presentation in this view WILL update Min Pres in the Main Planning and Cluster Set views, as well as in Sales Plan View
  • Note: Changing Receipt Units in this view WILL NOT update metrics in the Main Planning or Cluster Set views – this is by design, as receipts are suggested in JustEnough Assortment Planning via a complex algorithm considering sales and all user-specified constraints; because sales are considered in the output already, to then, go update Ttl Buy U would cause the sales plan to change, thus potentially invalidating the results of the receipt calculation. It WILL, however, update Receipt Units in Sales Plan view, as will changing Receipt AUC.
  • Calculations: Receipt plans are calculated via the JustEnough Receipt Planning Engine, which uses a complex algorithm to determine the optimal receipt flow based on sales and minimum presentation by period and user-defined constraints; because of this, system receipt recommendations require user action to recalculate, rather than recalculating on the fly as changes are made in the user interface. Recalculation can be done either for the selected placeholder/collection only by clicking Recalculate Receipts in this view, or for all placeholders/collections that require recalculation by clicking Recalculate in the Main Planning view header area.

  • Receipt Engine Outputs: the System Receipt Units and Ideal EOP Units metrics are outputs of the Receipt Planning Engine, which must be re-run when any of the inputs it considered change. These fields will be conditionally formatted any time a change is made that could invalidate the results of the last engine run – changes to sales, min pres, shrink, inventory adjustments, or receipt constraints all fall into this category.:

  • Plan Horizon vs. Receipt Planning Engine Horizon: if the Plan Horizon, set at the time of plan creation, extends beyond the receipt planning engine horizon for a given item, data that extends past the receipt engine horizon will appear in blue italics with zero values ($0.00, for example). See example in Plan Horizon vs. Replen Forecast Horizon below).
  • Replen Placeholders/Collections: when a placeholder/collection is of the Status Replen, the system pulls all relevant data to form its sales plan from the Forecasts module. Users should be aware of the following differences in this view when dealing with Replen items:
  • Read Only: all normally editable metrics in Receipt Plan view become read only, being populated directly from aggregated replenishment forecast data
  • Receipt Constraints: all receipt related constraints will be managed within the Forecasts module, thus receipt constraints in Assortment Planning for Replen placeholders/collections will be cleared; upon Status change to something other than Replen, the default constraints assigned to the corresponding product level, if any, will be reinstated
  • Receipt Periods: for Replen placeholders/collections, the necessary orders to support forecasted sales are generated from the Forecasts module and sent to Order Manager for processing, thus users don’t need to set Receipt Periods in this view the way they do for non-Replen items. Receipt Periods are set automatically based on when the generated orders are scheduled to arrive

  • Assortment Engine Lead Time: This setting determines the number of days forward from receipt date that received stock is assumed to be available to support sales
  • Default is 7 to assume receipts are available to sell one week after receipt
  • Can be adjusted based on a realistic assessment of the capabilities of your supply chain
  • When the engine is calculating receipts, this setting essentially determines the range of periods it must consider to meet the defined constraints for a given period; see examples for Periods of Supply and Min Stk to Sales below for details

  • Placeholder Cluster Constraints
  • Periods of Supply: user editable positive integer; sets the periods of supply (period type as defined by the current assortment plan) for the placeholder/collection instance as a minimum stock constraint for receipt engine outputs
  • Actual by period constraint is defined as the sales planned for this many periods forward from a given period
  • Note: this setting interacts with the system policy: Assortment Engine Lead Time (Maintain > Policies > Engine: Commitments)

  • Example: Assortment Engine Lead Time is set to 7 days, Periods of Supply is set to 2, and I am planning a receipt to arrive in Wk 1. For this constraint to be considered fulfilled for Wk 1, I must end the period with at least 60 units of inventory on hand to cover weeks 2 and 3. The lead time of 7 days stipulates that stock being received in given week cannot be used to cover the sales planned in that week, thus Periods of Supply begins calculating need at Wk 2 rather than Wk 1.

  • Periods Before Phase In: user editable positive integer; sets the default number of periods prior to Phase In date that initial receipt quantities for the placeholder/collection instance should be scheduled to be received
  • Example: I have a placeholder instance with a Phase In date of wk 4 and a Periods Before Phase In of 2 - my default calculated initial receipt for the placeholder instance will be in wk 2
  • Flow Cadence: user editable positive integer; sets the default number of periods between receipt dates calculated by the receipt planning engine
  • Example: I have a placeholder instance with a Phase In date of wk 3, a Periods Before Phase In of 2, and a Flow Cadence of 4 - if I have 3 receipts, they will occur in wk 1, wk 7, and wk 11. This is because the purpose of Periods Before Phase In is to ensure product can be in stores by the Phase In date, thus the first flow date is considered to be the Phase In date for the purposes of receipt engine calculation
  • Min Stk/Sales (Minimum Stock to Sales): user editable positive integer; sets the minimum stock to sales ratio for the placeholder/collection instance as a minimum stock constraint for receipt engine outputs
  • Actual by period constraint is defined as the sales planned for a given period multiplied by the constraint value (period sales of 10 * Min Stk/Sls of 4 = EOP stock requirement of 40)
  • Shrink % of Sales: user editable positive percentage to 2 decimal places (1.52%); sets the anticipated shrink units per period as a percentage of planned period sales units for the placeholder/collection instance as a minimum stock constraint for receipt engine outputs
  • Example: if I have a period with planned sales of 500 units and a Shrink % of Sales of 2.00%, I must have an EOP units of at least 10 for this constraint to be satisfied for the period
  • Target Sell-Through %: non-editable percentage (10.01%); S/T % for the placeholder/collection instance as defined in Cluster Set view
  • Calc Sell-Through %: non-editable percentage (10.01%); calculated sell-through percentage for the placeholder/collection instance, given the current sale and receipt plan shown in the view
  • Recalc Buy: user editable checkbox; sets whether System Receipt Units and Ideal EOP Units will be recalculated for the placeholder/collection instance when Recalculate Receipts in Receipt Plan view or Recalculate in the Main Planning view is clicked
  • Hold Receipts: user editable checkbox; sets whether the current total Receipt Units for the placeholder/collection instance can change to accommodate Placeholder Cluster Constraints upon recalculation
  • If unchecked, the system will calculate the receipt strategy to meet the specified constraints for as long as possible without increasing Receipt Units by moving receipt units forward in time
  • Note: If unchecked, there will be scenarios where the system will not be able to satisfy user defined constraints for the life of the item within the cluster
  • Receipt Plan Metrics
  • Site Count: non-editable integer; count of all sites the placeholder/collection is assorted to at whatever level the Pivot Grid is aggregated to for the period
  • Note: if multiple cluster set snapshots are effective during the life of the placeholder/collection, this number can vary by period
  • Sales Units: non-editable integer; planned/actual sales units for the period; read from Sales Units (user, not System) in Sales Plan view
  • Calculation: (Sales / AUR)
  • Minimum Presentation: user editable integer; Minimum units required to maintain floor presentation for each site selling the item
  • Calculation: From Receipt Plan
  • Note: the Mkdn Date metric in the Main Planning and Cluster Set views interacts with Min Pres to set the period in which the system should stop respecting it; if Mkdn date is set for the placeholder/cluster instance, Minimum Presentation will drop to zero from the corresponding period forward
  • BOP Units: non-editable integer; planned/actual beginning of period inventory units
  • Calculation: (Previous Period EOP)
  • Receipt Period: user editable checkbox; set the periods in which receipts can occur
  • If Flow Cadence is set, recalculating receipts will also check the appropriate Receipt Periods based on the defined frequency
  • Receipt Period will be checked for the period corresponding to the Periods Before Phase In Constraint but it will be un-editable
  • When receipts are recalculated, the engine output will respect Receipt Periods and only suggest receipts in periods that are checked
  • Users will not be able to enter Receipt Units in periods where Receipt Period is unchecked
  • Commit: user editable checkbox; sets the periods for which to create commitment orders when Create Commitments is clicked
  • Checked by default for checked Receipt Periods
  • Cannot be checked for periods where Receipt Period is not checked
  • Order By Date: non-editable date (10/2/2017); displays the order deadline for the corresponding Receipt Period/Units based on assigned default lead times; essentially, the first day of the receipt period minus Default Lead Days from the Main Planning view
  • Receipt Units: user editable integer; planned/actual receipt units for the period
  • Defaults to System Receipt Units unless user has manually entered a quantity and Applied
  • Calculation: System Receipt Units – Committed Units – On Order Units – On Order (Unknown) Units
  • Note: once a commitment order has been created (see Create Commitments, above), the quantity that was in this field will move into the Committed Units field, indicating that an order has been generated for the planned quantity; as the order moves through the commitment process, the units will transition or On Order Units (or On Order (Unknown) Units) and then ultimately show again as receipt units when the period in question actualizes, provided the units were actually received in that period; this ensures that units for the same order are not double-counted as they move through the process; see related field descriptions below for details
  • Note: once the above described transition occurs, it is still possible to add more Receipt Units to this field and place subsequent commitment orders for them as needed; this may happen automatically as your sales plan changes as a result of receipt plan recalculations but you can also simply enter the desired additional quantity in the field then again clicking Create Commitments when ready; each individual order will contribute to the totals shown and be tracked and transitioned through the applicable fields individually
  • Helpful Hint: entering Receipt Units at the Grand Total level in this view is a quick way to balance your receipt plan to the Ttl Buy U in the Main Planning and/or Cluster Set views if desired; simply aggregate the pivot grid to the desired level by removing fields as needed (ie. Cluster Set, Cluster, Style, etc.) and enter the Ttl Buy U value in the Grand Total column at the far right – the structure of your receipt plan, including flow dates and % to total of receipt units by period, will be respected while the system balances the period level entries to match the new Grand Total
  • Receipt AUC: user editable currency ($1.01); sets the average cost per unit for receipts in the period
  • Calculation: N/A
  • Receipt Cost: non-editable currency ($100); planned/actual cost of receipts for the period
  • Calculation: (Receipt Units * Receipt AUC)
  • Committed Units: non-editable integer; units for which commitment orders have been created but which do not have corresponding confirmed purchase orders
  • Auto-populated when commitments are created for periods having Receipt Units; read from commitment orders for the corresponding variant/cluster/period in Order Manager
  • Note: receipt units for a given period are transitioned out of Receipt Units to Committed Units when commitments are created for that period
  • Important: Commitment Units can be automatically decremented based on the presence of confirmed On Order Units but only if the system policy Assortment Order Type (Maintain > Policies > Module: Assortment) is set to Variant Orders and JustEnough has integrated with an ERP/POM system capable of returning order data with the UUID’s JustEnough generates for each order sent; if this is not the case, a custom procedure will need to be created to appropriately decrement Commitment Units and avoid duplication of units with On Order (Unknown) Units (where purchase order units the system is unable to tie to specific commitment orders populate)
  • On Order Units: non-editable integer; confirmed units on order for the period
  • Calculation: (purchase order units not yet received but scheduled for the period)
  • Note: As stated above in Committed Units, when the system policy Assortment Order Type (Maintain > Policies > Module: Assortment) is set to Variant Orders and JustEnough has integrated with an ERP/POM system capable of returning order data with the UUID’s JustEnough generates for each order sent, this field will populate with confirmed purchase order quantities by variant/cluster/period and decrement Committed Units accordingly; in the case where the system policy is set to Inventory Orders and/or external systems are NOT able to return UUID’s by order, purchase order units will appear in the On Order (Unknown) field
  • On Order (Unknown): non-editable integer; confirmed units on order for the period which cannot be matched to specific variant/cluster commitment orders
  • Calculation: (unlinked purchase order units not yet received but scheduled for the period)
  • Note: On Order (Unknown) units will be split across applicable clusters using the same splitting logic as discussed below in EOP Units
  • Important: See Committed Units above
  • Total Orders: non-editable integer; sum of all planned and confirmed incoming orders for the period
  • Calculation: (Receipt Units + Committed Units + On Order Units + On Order (Unknown) Units)
  • Total Orders AUC: non-editable currency ($1.01); average cost per unit of inbound orders for the period
  • Calculation: (Total Orders Cost / Total Orders Units)
  • Total Orders Cost: non-editable currency ($100); cost of inbound orders for the period
  • Calculation: (Total Orders Units * Total Orders AUC)
  • Stock Units to Sales Units: non-editable decimal (1.01) ; ratio expressing the relationship of BOP Units to Sales Units
  • Calculation: (BOP Units / Sales Units)
  • Shrink Units: non-editable integer; planned/actual shrink units for the period
  • Calculation: From Sales Plan view
  • Inventory Adjustment Units: non-editable integer; planned/actual inventory adjustment units for the period
  • Calculation: From Sales Plan view
  • EOP Units: non-editable integer; planned/actual end of period inventory units
  • Calculation: (BOP Units – Sales Units + Total Orders – Shrink Units + Inventory Adjustment Units)

  • Note: For placeholders/collections for which inventory exists in a DC, the EOP Units of the last actualized period will include a portion of available DC on-hand split out per the following logic:

    • Calculation: (Cluster Previous Period EOP Units + Cluster Future Total Orders – Cluster Future Planned Sales Units) / SUM(Cluster Previous Period EOP Units + Cluster Future Total Orders – Cluster Future Planned Sales Units for all clusters) * DC on-hand units

    • Note: While there is an inherent degree of potential inaccuracy in splitting DC on-hand inventory via this method, this was a conscious design decision to assume optimal distribution of available inventory based on future demand by cluster, as the alternative of including complete supply chain logic would have had a significant impact on the performance of the system
  • System Receipt Units: non-editable integer; system recommended receipt units for the period
  • Calculation: Receipt Planning Engine
  • Note: this metric seeds System Receipt Units in Sales Plan view as well
  • Ideal EOP Units: non-editable integer; system recommended EOP units for the period
  • Calculation: Receipt Planning Engine
