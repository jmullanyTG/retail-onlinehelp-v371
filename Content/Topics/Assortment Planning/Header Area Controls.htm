Header Area Controls

  • New button: Launches the Add Assortment dialog

  • Name: User editable alpha-numeric; sets the name for the assortment plan
  • Description (not required): User editable alpha-numeric; user description for the assortment plan
  • Calendar: User editable dropdown; sets the calendar for the assortment plan
  • Period: User editable dropdown; sets the lowest level period to be planned in the assortment plan
  • Product Hierarchy: User editable dropdown; sets the product hierarchy for the assortment plan
  • Level Contribution: User editable dropdown; sets the product hierarchy level at which location % contribution tables will be calculated
  • Site Hierarchy: User editable dropdown; sets the location hierarchy for the assortment plan
  • Important: cluster sets to be used within an assortment plan must be contained within a single leaf level of the assigned Site Hierarchy. For example, if you assign a hierarchy that goes down to Country level and contains the leaves North America, Canada, and UK, cluster sets within the assortment plan cannot have sites that exist in more than one country.
  • Linked Financial Plan (not required): User editable dropdown; sets a merchandise financial plan to which the assortment plan will be linked
  • Default Currency: User editable dropdown; sets the base currency to be used for all revenue metrics in the assortment plan
  • Home Site: User editable single-select tree dropdown; selects a site to use for determining the home currency VAT % to apply when calculating AUT/AUR within the assortment plan; populates from sites within the selected site hierarchy that have an assigned currency matching the selected Default Currency
  • Note: as VAT rates can fluctuate throughout the planning season, we’ve chosen to link this value to a site rather than using an explicit rate so that the Home VAT % will change dynamically as needed
  • Note: to accommodate cases where the site chosen as the Home Site closes or becomes otherwise inactive, this parameter will remain editable even after the user begins making changes within the assortment plan
  • Note: the system policy Allow DCs as Home Site, which can be accessed via Maintain > Policies > Module: Assortment, determines whether or not sites of type DC will appear for selection as the Home Site; this is to accommodate scenarios where planning occurs in a currency and/or country where there are no non-DC sites present
  • Warning: changing the home site in an existing plan will initiate a recalculation of all VAT dependent metrics within, such as AUT/AUR, etc.
  • Important: the site selected for the Home Site should mirror the currency and VAT % you will use as a basis for your planning activities
  • Sales Types: User editable multi-select dropdown; displays/sets the sales types used in the location % contribution calculations for the assortment plan
  • Replen AUT Price Type: User editable dropdown; displays/sets the price type from which AUT will be derived for placeholders of Replen Status
  • Replen AUR Price Type: User editable dropdown; displays/sets the price type from which AUR will be derived for placeholders of Replen Status
  • Replen AUC Cost Type: User editable dropdown; displays/sets the cost type from which AUC will be derived for placeholders of Replen Status
  • Period Horizon: User editable integer; sets the rolling number of weeks into the future that the assortment plan will extend
  • Created By: Non-editable alpha-numeric; name of the user who created the assortment plan
  • Created Date: Non-editable date; date the assortment plan was created on
  • Last Initialized: Non-editable date; date the Seed process was last completed for the assortment plan

  • Defaults button
  • Launches the Manage Assortment Defaults dialog for the current plan
  • Users can set default parameters for plan elements and functions used frequently within Assortment Planning so they don’t have to be manually set each time; default parameters can generally still be changed later by the user if needed
  • Defaults set here are applied to placeholders assigned to the specified level of the product hierarchy at the time of generation, whether manually within Line Visualization or through the Smart Start function
  • Note: changing defaults in this screen will not alter these settings for placeholders that already exist.
  • Defaults are set in an inheritance scheme, meaning defaults set at higher levels of the product hierarchy persist down through the levels beneath unless different defaults are set at those lower levels as exceptions
  • Example: my hierarchy has a department called Men’s Tops, and beneath Men’s Tops exist the sub-departments Men’s Tees, Men’s Wovens, and Men’s Knits. I set a default at the Men’s Tops level then set a different default at the Men’s Knits level. Men’s Tees and Wovens inherit the same default set at the Men’s Tops level, but Men’s Knits uses the default set at it’s own level.
  • All default settings are not required

For this constraint to be considered fulfilled for Wk 1, I must end the period with at least 60 units of inventory on hand to cover weeks 2 and 3. The lead time of 7 days stipulates that stock being received in given week cannot be used to cover the sales planned in that week, thus Periods of Supply begins calculating need at Wk 2 rather than Wk 1

  • Note: all available size curves populate for selection in this field.

  • Example: I have a placeholder instance with a Phase In date of wk 4 and a Periods Before Phase In of 2 - my default calculated initial receipt for the placeholder instance will be in wk 2
  • Flow Cadence: user editable positive integer; sets the default number of periods between receipt dates calculated by the receipt planning engine; used in the Receipt Plan supporting view of Line Visualization
  • Example: I have a placeholder instance with a Phase In date of wk 3, a Periods Before Phase In of 2, and a Flow Cadence of 4 - if I have 3 receipts, they will occur in wk 1, wk 7, and wk 11. This is because the purpose of Periods Before Phase In is to ensure product can be in stores by the Phase In date, thus the first flow date is considered to be the Phase In date for the purposes of receipt engine calculation
  • Min Stk/Sales (Minimum Stock to Sales): user editable positive integer; sets the minimum stock to sales ratio for placeholder instances as a minimum stock constraint for receipt engine outputs within the Receipt Plan supporting view of Line Visualization
  • Actual by period constraint is defined as the sales planned for a given period multiplied by the constraint value (period sales of 10 * Min Stk/Sls of 4 = EOP stock requirement of 40)
  • Shrink % of Sales: user editable positive percentage to 2 decimal places (1.52%); sets the anticipated shrink units per period as a percentage of planned period sales units for placeholder instances as a minimum stock constraint for receipt engine outputs within the Receipt Plan supporting view of Line Visualization
  • Example: if I have a period with planned sales of 500 units and a Shrink % of Sales of 2.00%, I must have an EOP units of at least 10 for this constraint to be satisfied for the period
  • Maximum Pack % Total: user editable positive percentage to 2 decimal places (60.00%); defines the maximum % of units in a commitment order of the units within prepacks can represent as a constraint for the prepack optimization engine within Order Manager
  • Note: when set to less than 100%, units not placed in prepacks will be fulfilled as single units
  • Example: I have two placeholder instance orders for wk 4 for 500 units each; I choose to prepack them together and have a Maximum Pack % Total of 60.00% - the maximum number of units existing in prepacks for this order is 600, however, my prepack output could be less than 600 if the prepack configurations being used don’t evenly roll up to 600 units based on the need calculation by site
  • Pack Tolerance: user editable positive percentage to 2 decimal places (5.00%); defines the number of units over the total order quantity the engine may suggest in an attempt to fulfill the Maximum Pack % Total, expressed as a percentage of the total order quantity; used as a constraint for the prepack optimization engine within Order Manager
  • Example: I have a total order quantity of 1,000 units to be prepacked, a Maximum Pack % Total of 100%, and a Pack Tolerance of 1.00% - the engine may suggest an adjustment of the total order quantity of up to 1010 units to allow for 100% of the original order quantity to be fulfilled by units in prepacks.
  • Note: when calculating prepack optimization against a Pack Tolerance, the engine will prioritize stores with greater need first
  • Maximum Packs Per Location: user editable positive integer; sets the maximum number of prepacks of any configuration that can be allocated to a single site within the prepack optimization calculation to fulfill the anticipated sales for that site; used as a constraint for the prepack optimization engine within Order Manager
  • Example: the prepack optimization engine determines that a site requires 50 units from an order being prepacked, Maximum Packs Per Location is set to 5, and the selected prepack configurations all contain 9 units - ignoring site need by size for this example, the engine can only use 5 prepacks, containing 45 units total, as the prepack component of order fulfillment for the site and must assume the remaining 5 units will be fulfilled as single units.
  • Underage Weight: user editable positive decimal to 2 places (1.25); sets the relative preference for underages when calculating optimal unit allocation by site; used as a constraint for the prepack optimization engine within Order Manager
  • Note: used in conjunction with Overage Weight and Pack Weighting and calculated as a % to total of the 3 parameters (ie. all 3 are summed then divided by the number of constraints that were not NULL - thus, if only 2 of the 3 are set, the sum of their values is divided by 2 rather than 3 to determine the % to total).
  • Overage Weight: user editable positive decimal to 2 places (1.25); sets the relative preference for overages when calculating optimal unit allocation by site; used as a constraint for the prepack optimization engine within Order Manager
  • Note: used in conjunction with Underage Weight and Pack Weighting and calculated as a % to total of the 3 parameters (ie. all 3 are summed then divided by the number of constraints that were not NULL - thus, if only 2 of the 3 are set, the sum of their values is divided by 2 rather than 3 to determine the % to total).
  • Maximum Overage: user editable positive integer; sets the number of units per site the prepack optimization engine may overallocate during calculation to reach an acceptable solution; used as a constraint for the prepack optimization engine within Order Manager
  • Maximum Overage %: user editable percentage to 2 decimals (1.25%); sets the percent of total need in units per site the prepack optimization engine may overallocate during calculation to reach an acceptable solution; used as a constraint for the prepack optimization engine within Order Manager
  • Pack Weighting: user editable positive decimal to 2 places (1.25); sets the relative preference for using the fewest prepack configurations possible when calculating optimal unit allocation by site; used as a constraint for the prepack optimization engine within Order Manager
  • Note: used in conjunction with Underage Weight and Overage Weight and calculated as a % to total of the 3 parameters (ie. all 3 are summed then divided by the number of constraints that were not NULL - thus, if only 2 of the 3 are set, the sum of their values is divided by 2 rather than 3 to determine the % to total).
  • Minimum Units Per Pack: user editable positive integer; sets the minimum number of units the prepack optimization engine may suggest in single prepack configurations; used as a constraint for the prepack optimization engine within Order Manager
  • Note: only valid when Optimize Prepack is checked for a given prepack group, otherwise user selected prepack configurations are used
  • Maximum Units Per Pack: user editable positive integer; sets the maximum number of units the prepack optimization engine may suggest in single prepack configurations; used as a constraint for the prepack optimization engine within Order Manager
  • Note: only valid when Optimize Prepack is checked for a given prepack group, otherwise user selected prepack configurations are used
  • Num Pack Configs: user editable positive integer; sets the maximum number of configurations the prepack optimization engine may suggest when determining optimal fulfillment of need within an order; used as a constraint for the prepack optimization engine within Order Manager
  • Note: only valid when Optimize Prepack is checked for a given prepack group, otherwise user selected prepack configurations are used

  • Seed button: Launches the Seed Placeholders dialog

  • Purpose: because of the way placeholder entities are used within Assortment Planning, a necessary step in creating new plans is generating equivalent placeholder entities for existing items for which you want to utilize historical selling information
  • Note: ideally, this process will only be done one time per plan, as all new items created in Assortment Planning are generated as placeholder entities, however, the process can be re-run to pick up new items that weren’t present at the time of seeding without adversely affecting anything that has already been seeded – the process simply looks for variants that don’t have placeholder entities created and creates them
  • Seeding and Attribute Inheritance: when seeding placeholders from historical variants, any attributes attached to those variants will also be attached to the resultant placeholders, provided the following are true:
  • A valid attribute set is associated with the product level to which the variants belong; association can be by directly choosing an attribute set for that level of the product hierarchy or through inheritance of a set from a level above
  • That attribute set is assigned to both the Variant and Variant Placeholder entity types
  • That attribute set includes all attribute families containing the attribute values associated with the variants placeholders are being created for
  • All applicable attribute families are assigned to both the Variant and Variant Placeholder entity types
  • All attribute values associated with the variants placeholders are being created for are marked as Allowed for the corresponding attribute set
  • See Attribute Management Standard Documentation for details on creating and assigning attribute sets, families, and values
  • Start Date: user editable date picker; sets the historical date you want the system to begin at when generating placeholders for existing items
  • Note: this should be set to the farthest date back for which you want to utilize historical sales in Assortment Planning – a general best practice is at least 2 years, as most businesses typically view data this far back as being relevant to current plans
  • Important: Start date cannot fall before the date designated by the current date minus the number of days specified in the Assortment days history to maintain system policy, as noted above in Prerequisites > Review/set foundational system policies; this policy can be accessed via Maintain > Policies > Module: Assortment
  • Execute: click after selecting a Start Date to populate the Assortment Seed Placeholders section below
  • Selected: user editable checkbox; selects the cluster sets in which the historical items you wish to generate placeholders for were present
  • Use Default Cluster Set(s) per product level: this option appears at the top of the cluster set list and has been included to allow users to easily seed historical placeholders for the default cluster sets assigned by product level.
  • Error: For this option to work properly, at least one default cluster set must be assigned, either directly or through inheritance from a level above, for all product levels present in the plan being seeded; an error will be raised indicating the need to do this if it has not been done at the time seeding is initiated
  • Error: this option cannot be used in tandem with the selection of other cluster sets within the list; if you attempt to do so, an error will be raised
  • Note: because of the dynamic nature of cluster configurations, all that is needed here is to select the cluster sets that contain the sites where the items sold historically – sales assignments are made at site level and returned based on the active cluster set snapshot for a given period
  • Cluster Set: non-editable alpha numeric; description for each validated cluster set within the current plan
  • Overlapping Cluster Sets: non-editable alpha numeric; description for any validated cluster sets that overlap with a given cluster set – meaning it contains some or all of the same sites as another set
  • Note: while it is possible to have overlapping cluster sets within a single plan, it is not possible to assort placeholder items to multiple overlapping cluster sets – you must choose one cluster set per group of sites in which to plan
  • Current Placeholder Count: non-editable integer; count of all placeholder items currently generated that registered sales in a given cluster set from the Start Date forward
  • Fields: opens the fields Customization dialog; users can drag fields to/from the Assortment Seed Placeholders column header area and this dialog to customize the view
  • Custom layouts can be saved/restored by right-clicking in the Assortment Seed Placeholders area and selecting Save Layout or Restore Layout, respectively

  • Cancel: unchecks selected cluster sets in the Assortment Seed Placeholders section
