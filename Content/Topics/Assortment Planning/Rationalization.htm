Rationalization

The Rationalization calculation is meant to give users an idea of, for a historical period of time, what the most efficient number of variants would have been in order to reach a specific financial target. The primary mechanism we use for this is a marginal return curve. What we’ve found across retailers, product types, time and geography is that, given a large enough sample, these marginal return curves are extremely stable and repeatable. It is possible to make modest accuracy gains by modelling the distribution of sales within an insular set of product and stores with a Poisson distribution, but for ease of both implementation and calculation, the marginal return curve is close enough. The following steps are part of the Rationalization process:

  • Data Selection
  • The explicit inputs for creating a Rationalization Scenario are the Target Segments, each being defined by a Target Filter, (optional) Supplemental Filter, Sales Basis (units/revenue), Sales Type(s) and Cluster Set selection. Scenario level selections include Scenario Target, (optional) Plan Basis, and Window of Time. Implicit selections include the period type assigned to the assortment plan and minimum thresholds see Data Cleansing below).
  • If Supplemental Filter is set for a segment, the system uses this in place of the Target Filter in order to calculate marginal return curves; sales data is then aggregated to variant level (style/color) by site cluster and by period.
  • Data Cleansing & Deseasonalization
  • Before calculating, several steps are taken to remove anomalies from the data that would skew subsequent calculations. The most common causes of inaccuracies when using historical sales to calculate future projections via a simple average are:
  • The item wasn’t available for sale during part of all of the window of time
  • If averaging the entire window of time, zeros may be included
  • If averaging non-zero periods only, those values may not be representative of a true average period during the window of time
  • The item had inventory levels low enough to negatively impact sales
  • During this phase, the system determines which variants in the data set were considered “active” during the specified comparison time frames for each segment. The following system policies (located in Maintain > Policies > Module: Assortment) exist to allow users to fine tune the output of this phase: This is accomplished through the following steps:
  • Remove Outliers: prior to deseasonalizing the data, we first remove any extreme outliers on the low end of the data set, as these would artificially deflate the deseasonalized average we use in subsequent steps. To do this, we remove a % of each variant’s maximum sales
  • System Policy: Average Inclusion Threshold; default set at 3%
  • Deseasonalize: data is then adjusted by period, flattening the sales trend across the lifecycle of each variant to allow for better comparison across time; this process is referred to as de-seasonalization. To achieve this, we sum variant sales by period, calculate the median total sales across all periods and replace any period totals which fall below a system policy defined % of the median, calculating a period to mean for each period based on the revised values, then, finally, dividing each period’s revised sales by the period to mean
  • System Policy: Median Period Total Threshold; default set at 50%
  • Remove Abnormal Data Points: after deseasonalization, we calculate the average period sales for each variant then remove any periods that fall below a system policy defined percentage of this average. This is done to account for periods where we can reasonably infer that the item’s inventory was no longer allowing an accurate gauge of its sales potential;
  • System Policy: Active Variant Threshold Factor; default set at 30%
  • Note: this means we do not remove outliers with sales that are much higher than the rest of the dataset – this was an intentional decision to maintain consistency with logic used elsewhere with the JustEnough suite of applications, including the Rate of Sale Calculator
  • Remove Variants With Insufficient Periods Remaining: after all anomalous periods have been cleansed, we calculate the number of periods remaining for each variant and remove variants without remaining periods greater than or equal to a system policy defined percentage of the window of time in the scenario.
  • System Policy: Active Variant Time on Offer Minimum; default set to 31%
  • Example:  given a period type of week and a window of time of one month, setting this policy to 31% will eliminate any site level/variant that does not have at least 2 active selling periods registered
  • Building Marginal Return Curves
  • Once data has been cleansed, the system calculates cluster/variant productivity by taking the average across the remaining periods. From here, we calculate two different marginal return curves.
  • Inter-style Marginal Return Curve: The most important curve is the inter-style marginal return curve (MRC).
  • In order to calculate, the system orders productivity values for all variants from greatest to least, and divides each Nth entry by the (N-1)st entry, defining N = 1 as 1, to get the marginal value of each variant to the cluster.
  • When the target productivity is more than the productivity of the active variants, the marginal return curve must be extended. We test for this by determining the ratio of the target productivity divided by the average productivity of the active variants – if the ratio is less than or equal to 1, the curve does not require extension. If the ratio is greater than 1, we extend the curve by creating an incremental marginal return curve (IMRC), and then using the increments defined therein to determine the marginal return for variants beyond the number of currently active variants. The new IMRC is assumed to take the form of an exponentially decreasing function that is fitted using characteristics from the original marginal return curve.
  • Note: this may artificially overstate the marginal productivity of additional variants added to a segment. Originally, the extension was to hold the last calculated marginal value to project the curve out, but numerous situations were found where the final two values were very close in magnitude, causing very small subsequent marginal values. In general, this calculation is intended to be somewhat conservative when projecting over unknowns.
  • Intra-style Marginal Return Curve: The intra-style curve is a supplemental curve used in tandem with the Inter-style MRC to assist in the determination of how placeholder items created during Smart Start should be grouped into collections (ie. This crew neck tee in 5 colors, this V-neck tee in 3 colors, and so on).
  • In this case, the system indexes each variant according to its rank within its style (with the most productive variant ranking 1, the second ranking 2, etc.).
  • The system then calculates each style’s MRC in the same manner as the Inter-style MRC, except it does not project beyond the calculated values.
  • Once each style’s curve has been calculated, they are combined by taking the marginal value of each variant with similar indices and weighting their contribution by their productivity values.
  • When an index is reached for which there are no values, the same projection procedure used for Intra-style MRCs is used with the following difference: while the Inter-style MRC is entirely recalculated for all variants when extension is needed, the Intra-style curve retains the marginal return increment for existing variants and the exponential curve is only used for the additional variants that necessitated the extension
  • Note: if Collection Option is not set to Multiple Variants Per Collection, the Intra-style MRC is not needed and is, therefore, not calculated
  • Marginal Return Curve Extrapolation: this system policy, accessible in Maintain > Policies > Module: Assortment, sets the number of data points required in a given period for the system to use the original calculated intra-style marginal return curve in determining the number of recommended choices to assign to each recommended collection. If this minimum is not met, the system will calculate a weighted curve based on the original intra-style marginal return curve and a curve calculated at a higher level of the product hierarchy to provide a more accurate recommendation.
  • Determining Recommended Variant Count
  • The next step in the process is to generate the output of recommended variant count for each segment/cluster.
  • First, the system calculates LY Variant Productivity, expressed as the sum of the Active Variant productivity numbers
  • In conjunction with the Inter-style MRC, the system builds an Expected Volume Curve, which, given the count of LY Active Variants and the LY Variant Productivity, determines the expected volumes for each other variant count. The system calculates the expected volume for the Nth variant count in the curve by taking the LY Variant Productivity and multiplying by the ratio of the entry corresponding to the LY Active Variants in the Inter-style MRC, and the Nth entry.
  • Example: if LY Active Variant count is 5, and LY Variant Productivity is $1,000, in order to calculate the expected volume for 4 variants, the system calculates the ratio of the 5th to 4th entries in the Intra-style MRC, and multiplies by $1,000.
  • Once the Expected Volume Curve has been generated, the recommended Variant Count is set equal to the greatest entry for which the expected volume is less than or equal to the TY Act New Sales volume for each segment/cluster.
  • Determining Collection/Variant Assignments
  • Once the Interstyle and Intrastyle marginal return curves are calculated and the choice count has been determined, the final step is to determine the number of collections and the number of variants in each collection
  • Assuming the user has selected the Collection Option Multiple Variants Per Collection, a marginal return matrix is calculated, in which each entry is a product of the Interstyle and Intrastyle curves
  • The top N entries in the matrix are then chosen where N=choice count. The chosen entries then represent the number of collections and variants per collection. Each row represents a collection and each column represents a variant within that collection
