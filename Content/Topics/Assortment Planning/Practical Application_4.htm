Practical Application

Promotions planning is designed to give the user a great deal of flexibility when modeling promotional effects for products within your plan, however, there are best practices you should adhere to in order to maintain a high level of efficiency and get the most out of this area of Assortment Planning:

  • Using Defined Promotions as Templates – because promotions, in and of themselves, do not have ties to specific products or levels of the site hierarchy, they can be used over and over for other areas of your business to which you want to apply the same basic promotion

 

Note in the above example, the same base promotion has been applied to three distinct product areas, including different cluster sets, start and end dates, and even with different Final Indices and Hold Sales settings. The same AUR Effect and Behavior will be applied to all applications, but aside from that, as long as applications occur within the Eligibility Start and End Dates (or the Is Always Eligible option is checked), the user retains a great deal of flexibility with the specifics of how the promotion is applied to each area.

 

This same methodology is mirrored to an extent within the details of each application as well. Start and End dates, Final Indices, Hold Sales options, and even participation in the promotion application can be varied at the placeholder/cluster level. Making use of the ability to make changes at the application and detail levels will help minimize the number of promotions users need to create and maintain.

  • Using the Is Always Eligible Option – If you have one or more standard promotions you tend to use often, setting the up using the Is Always Eligible option is an easy way to ensure they are always available for application.

 

Suppose you find yourself running 20% off promotions frequently as you plan. In the above example, you simply create one promo with a 20% off Promotion Discount % and the Is Always Eligible option checked and you will be able to apply that promo with different Final Indices to different products and site hierarchy levels without ever have to specify or extend Eligibility Start or End Dates or create duplicate promotions to achieve the same effect.

  • Using Promotional Metadata for Product Filter Creation – Information about promotions created within Assortment Planning can be used as a quick way to create product filters for the items to which a particular promotion or promotion application has been applied.

 

To do this, simply select one of the available fields beneath the Assortment Promotions heading in the Filter Condition dialog:

 

 

  • Promotion Application: defines the filter based on the variants included in a specific promotion application(s)
  • Conditions
  • Equals: allows the user to select one from the  promotion applications in the current assortment plan
  • Contains the Phrase: allows the user to specify text contained anywhere within the desired promotion application description(s)
  • Contains the Phrase at the Beginning: allows the user to specify text contained at the beginning of the desired promotion application description(s)
  • Contains the Phrase at the End: allows the user to specify text contained at the end of the desired promotion application description(s)
  • Is One of the Following: allows the user to select many from the  promotion applications in the current assortment plan
  • Note: promotion application descriptions are system generated in the following format: (Promo Name): (Product Filter Name) / (Cluster Set Name(s)) / (Promo Application Start Date) – (Promo Application End Date)
  • Example: My Test Promo: Mens Graphic Tees / North America Clusters / 2017/03/01 – 2017/06/01
  • Promotion Name: defines the filter based on all variants included in ANY promotion application of the selected promotion(s)
  • Conditions: same as above but with Promotions and not Promotion Applications
  • Product Filter: defines the filter based on all variants contained in the selected product filter for which there are promotion applications
  • Conditions: same as above but with Product Filters
  • Note: selectable Product Filters are only those that have promotion applications against them
  • Cluster Sets: defines the filter based on all variants assorted to the selected Cluster Set(s) for which there are promotion applications
  • Start Date: defines the filter based on all variants having promotion applications that begin relative to a selected start date(s)
  • Conditions
  • Equals: allows the user to specify a single Start Date for promotion applications to include
  • Is After: allows the user to specify a single date as a starting point; all promotion applications beginning after the date selected will be included
  • Is After or On: allows the user to specify a single date as a starting point; all promotion applications beginning on or after the date selected will be included
  • Is Before: allows the user to specify a single date as an ending point; all promotion applications beginning before the date selected will be included
  • Is Before or On: allows the user to specify a single date as an ending point; all promotion applications beginning on or before the date selected will be included
  • Is Between: allows the user the specify a start and end date as a range; all promotion applications beginning between the start and end dates specified will be included
  • Is One of the Following: allows the user to specify multiple Start Dates for  promotion applications to include
  • End Date: defines the filter based on all variants having promotion applications that end relative to the selected end date(s)
  • Conditions: same as above but with End Dates instead of Start Dates
  • Placeholder is on Active Promotion: there is another related Promotions parameter that can be used in conjunction with the fields above to return only items that are on Active promotion applications. This parameter can be found in the (Basic Variant Placeholder Information) section of the Filter Condition Fields:
