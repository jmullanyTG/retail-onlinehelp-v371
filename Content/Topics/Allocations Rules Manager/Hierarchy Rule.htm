Hierarchy Rule

The hierarchy rule chooses the data to aggregate based on the items selected. Users will select the level to which they wish to aggregate (i.e. you do not specify the specific class such as men’s knits, in this case we specify that it runs at class).

 

Users would first select the items which they intend to allocate.

Stores are selected at the time of allocation. I.e. when you actually execute the allocation, it uses the stores that you select in the wizard.

In this example if there is no size component, i.e. we chose to aggregate only to the class level and not to the class size level.

If, in this example, we were to change the rule to aggregate to class and size, the tool will now aggregate the sales to a class (women’s denim) and size (25, 26, 27, 28).

You will notice that the allocation is done per individual item and therefore the contributions of each store could be different by size.

 

Advantages

  • Hierarchy Rules can be used to allocate multiple sizes of products with the same rule allowing for different contributions by store
  • Hierarchy rules can be used regardless of the sizes to allocate different products to stores based in the different products classes contribution
  • This all means that you only use a single rule to allocate instead of having to create multiple fixed rules

Disadvantages

  • Because the allocation has to run for every item individually, it is slower than a fixed rule
